# 案例：AI 辅助协调决策引擎
### ——从"会议协调"到覆盖七类真实工作场景的可扩展体系

---

## 一、问题背景

在大型组织中，围绕"领导/关键角色时间"的协调工作是一类看似琐碎、实则极耗人力和判断力的工作。它以两种典型形态出现——一种是**多方凑时间**，一种是**多人抢时间**。以我曾负责的场景为例：

### 场景一：大型评审会的多方日程冲突

某类技术评审会要求参会专家出席率达到 70% 以上方可成席。实际操作中：

- 每位专家每天日程繁忙，且各自冲突时间不同，需要逐一询问、逐一确认；
- 部分专家消息回复不及时，或正在开会无法响应，需要电话催问；
- 涉及高层领导时，通常不直接询问领导本人，而是通过其秘书确认时间，且沟通方式、措辞、催办分寸都有一套不成文的规矩；
- 若某个时段冲突专家过多，需重新选择时段、重新逐一询问，可能反复多轮才能凑齐 70% 出席率；
- 若必须配合领导的时间，则其他专家的时间只能围绕领导时间来协调。

**场景 B：临时改期或取消**

会议临近开始甚至已经开始时，领导因临时冲突要求改期或取消，此时前述的整个协调流程需要重新走一遍——在人人日程都很满的大厂环境中，这种"推倒重来"的成本极高。

**现有的人工解决方案**：将会议尽量安排在非工作时间（早 8 点前、晚 10 点后）以规避冲突；若参会者中有高职级领导，则优先服从领导时间；日常协调工作主要依赖行政/项目助理逐一沟通、判断、汇报。

这项工作看似是"排期"，实际考验的是**沟通礼仪的分寸感、组织规则的隐性知识、以及何时该自主处理、何时该向上升级请示的判断力**。

### 场景二：员工个人约领导时间的冲突仲裁

除了大型评审会，秘书日常还要处理另一类结构不同的难题：多位员工分别、独立地来申请占用领导的时间。这与场景一的本质区别在于——场景一是"一场会议需要凑齐多少人"，场景二是**"多个互不相关的需求，在争抢同一份稀缺资源"**，是一个资源仲裁问题，而非日程匹配问题。实际操作中常见的情况包括：

- 多位员工几乎同时来问领导某个时段是否有空，时间上可能自然错开，也可能互相冲突；
- 领导的行程本身已经非常满，但员工的事项又比较急，一定要拿到一个明确的时间；
- 员工之间互不知情，都认为自己的事最紧急，秘书需要独自判断谁该优先，且很难对外解释清楚排序依据；
- 有些申请事项主题接近（比如都是同一个项目的不同侧面），但员工各自单独来问，秘书需要意识到"这些其实可以合并成一次沟通"；
- 如果单纯按"谁先来问"排序，容易出现真正紧急的事被排在后面；如果单纯按职级或声音大小排序，又容易打击长期被压后的合理需求。

这类场景对协调者的考验，不是"信息收集"，而是**在信息不对称、资源确定稀缺的情况下做排序和取舍**——这恰恰是通用日程工具完全不涉及的能力。

## 二、为什么现成的 AI 工具解决不了

目前无论是通用日程助手，还是企业协作平台内置的 AI Agent（如飞书 aily、OpenClaw 类产品），核心能力都集中在"信息收集与呈现"层面：读取日历、匹配空闲时间、生成会议邀请、整理会议纪要。这类能力已经相当成熟，且平台化程度很高，个人再去重复实现同类工具，价值有限。

但这些工具普遍缺失三类能力，而这三类能力恰恰是本场景中最难、最耗人力的部分：

| 缺失能力 | 具体体现 |
|---|---|
| **角色权重判断** | 不同参会人在日程协调中的"话语权"不同——普通专家可直接询问，领导需经由秘书间接确认，且措辞分寸不同 |
| **有效性判定** | "出席率达标"不等于"会议有效"——组织者、主席、关键领域专家缺席会导致评审结果无效，这条规则因会议类型而异 |
| **升级时机判断** | 常规冲突可由协调者自主处理，但反复约不到、或必须动用非工作时间时，需要明确的信号触发向上汇报 |

换句话说，现成工具解决的是"执行层"问题，而本案例要解决的是**"决策策略层"**问题——如何把这套原本存在于资深助理脑中的隐性规则，转化为一套可配置、可复用的系统逻辑。

## 三、场景一解决方案：一套可配置的"会议协调决策层"

整体设计思路是：**让 AI 承担高频、机械、耗时的执行工作（问询、催办、匹配、记录），把稀缺的判断力集中在真正需要人介入的少数关键节点上。** 具体拆解为四个模块。

### 模块 1：分级沟通引擎

针对不同角色，预设不同的沟通策略，而非统一的"发消息问时间"：

| 角色类型 | 沟通方式 | 措辞策略 | 催办规则 |
|---|---|---|---|
| 普通参会专家 | 直接消息 | 公事公办，简洁说明会议背景与紧急程度 | 超过设定时长（如 2 小时）无响应 → 自动电话提醒；若对方正在开会 → 待其会议结束后自动回拨 |
| 关键领域专家 / 主席 | 直接消息 + 标注"关键角色"提示 | 同上，但需明确告知其缺席将导致会议无效 | 响应优先级更高，无响应即触发升级提醒 |
| 高层领导 | 经由秘书间接确认 | 需说明会议背景、紧急程度、召集人及参会范围，询问领导可行时段及期望反馈时限，语气更正式、留有余地 | 不由 AI 直接催问领导本人，仅可催问秘书 |

这套规则可在会议创建时一次性设定（适用于定期例会），或按临时会议单独说明，后续无需重复配置。

### 模块 2：多时段并行匹配

摒弃"先提一个时段、冲突了再问下一个"的串行模式，改为**一次性推送多个候选时段**，由各参会人（或其个人 Agent）并行响应：

- 系统读取参会人已有日程，自动标注每个候选时段状态：空闲（✅）、已知冲突及冲突事由（⚠️，供本人判断是否可调整）、已接受的重要会议（❌，视为不可用）；
- 各方响应实时汇总为一张协调看板，按"当前满足出席率的候选时段"排序展示；
- 若所有候选时段均不满足 70% 出席率或关键角色缺席，系统自动发起下一轮候选时段征询，无需人工重新启动流程。

### 模块 3：有效性判定与例外规则

出席率是硬性制度要求，不可造假，但允许"委托代理"（专家可指派他人或未来的"专家分身" Agent 代为出席）。系统需内置的判定逻辑：

- 出席率 ≥ 70% 且组织者/主席/关键专家在场（或已委托代理）→ 会议有效，可自动确认；
- 出席率达标但关键角色缺席且无代理 → 判定无效，自动提示需重新协调，不进入人工审核环节；
- 该规则集在会议类型创建时预设一次，后续同类会议自动复用。

### 模块 4：升级机制（人机分工的核心）

这是整套系统里最关键的设计——明确划定"AI 可以自主处理"与"必须交由人判断"的边界：

- **AI 自主处理**：常规时段征询、催办、多轮匹配、出席率与有效性的规则判定、会议确认后的日程创建、会中纪要与待办生成、会后遗留问题跟踪与提醒；
- **升级给人**：多轮征询后仍无法满足出席率或关键角色到场要求，且已尝试常规工作时间范围内的可选方案 → 系统整理清晰的"冲突摘要 + 候选方案"提交给协调者，由协调者判断是否将会议移至非工作时间，并据此向领导汇报，由领导最终拍板。

人在这里的角色，不再是"逐一问时间"的执行者，而是"看结论、做判断、担责任"的决策者——这也是这套系统真正释放助理生产力的地方。

### 前瞻延伸：个人 Agent 分身与会议参与方式的演变

随着"个人 Agent 分身"的发展，未来每位职场人或都可拥有代表自己参会、实时同步会议进展、仅在涉及自身关键决策点时才需要真人介入的分身系统。届时，"是否所有人都要全程参会"这一前提本身可能被重新定义——会议冲突问题的解法，可能不再是"更聪明地排期"，而是"减少对全员实时在场的依赖"。这是本方案在设计时预留的演进方向，也是判断这类系统长期价值时值得关注的趋势。

## 四、场景二解决方案：一套"稀缺资源仲裁层"

场景二的核心不再是"匹配空闲时间"，而是**在多个申请之间排序、并在资源饱和时给出可执行的替代方案**。延续同一套"AI 自主执行、人做关键判断"的设计哲学，具体拆解为以下规则模块：

### 规则 1：优先级评分

不能简单按"先到先得"或"谁职级高"排序，需要一套可量化、可配置的评分维度：

- 事项紧急程度（是否有明确外部截止时间，如客户、审计、上级交办）；
- 事项本身的决策重要性（是否是跨部门卡点、是否阻塞后续工作）；
- 已等待时长的修正加权（避免长期被压后的合理需求永远排不上队）。

### 规则 2：同主题合并建议

系统识别出多个申请主题相近时（例如都与同一项目相关），主动建议将其合并为一次沟通，减少领导时间被碎片化占用，也减少重复沟通成本。

### 规则 3：日历饱和时的替代方案

当领导日程确实排满、无法插入新会议时，系统不直接回复"没有时间"，而是给出可执行的替代路径：建议压缩会议形式（如改为简报审阅 + 简短确认）、给出预计等待时长，而不是让申请无限期悬置。

### 规则 4：透明度反馈

对暂时排后的申请人，系统给出脱敏后的排序依据说明（如"当前有更紧急的客户事项优先处理，预计您的事项将在 X 小时内安排"），减少协调者被反复追问的成本，也降低"凭什么排我后面"的不满情绪。

### 规则 5：升级机制

当出现两个申请优先级得分相近、且日历确实排不下时，系统停止自主排序，整理清晰的对比摘要（事项、紧急度、已等待时长）提交给协调者，由人来做最终取舍——这与场景一的升级逻辑完全一致：**执行和常规排序交给 AI，真正的两难判断留给人**。

场景一与场景二虽然问题形态不同（多方凑时间 vs. 多人抢时间），但底层设计逻辑是同一套框架：分级沟通/评分规则 → 自动化匹配/排序 → 有效性/优先级判定 → 明确的升级边界。这也说明这套框架不是针对单一问题的一次性方案，而是可以迁移到同类"稀缺资源协调"场景的通用设计模型。

## 五、支持材料

为验证上述设计的可用性，配套制作了以下材料：

- **决策流程图**：可视化场景一从征询发起到升级判断的完整链路（含循环与升级分支）；
- **协调看板原型（v2）**：模拟场景一的多时段并行匹配看板，包含推荐时段排序、响应进度、冲突可协调性标注、征询历史记录、代理出席标注等功能；

## 六、预期效果

以真实经历估算：一场需要 70% 出席率、涉及 10+ 位专家及 1 位高层领导的技术评审会，传统协调方式往往需要 2-3 轮消息 + 电话往返，耗时约 4-8 小时（跨多天），且改期需重新走一遍全部流程。

通过上述"分级沟通 + 并行多时段匹配 + 规则化有效性判定 + 明确升级机制"的设计，理论上可将协调者的直接工作量压缩至：

- 首次配置规则（约 10-15 分钟，且可复用于同类会议）；
- 处理系统升级上来的少数例外情况（通常 0-2 次/场）；
- 其余问询、催办、匹配、确认、纪要、待办跟踪均由系统自动完成。

场景二同理：日常多个员工申请领导时间的场景，秘书原本需要凭个人经验反复权衡、逐一沟通解释排序理由，每天可能耗费零散但持续的精力。通过评分排序、合并建议与自动化的透明反馈，协调者的介入可收窄至：处理评分相近的两难升级情况，以及审核系统给出的合并/替代建议是否合理。

对协调者而言，无论哪个场景，工作重心都是从"重复沟通与信息传递"转移到"规则设计与例外决策"——这正是助理岗位在 AI 时代最应该保留、也最有议价空间的部分。

## 八、体系化：从两个场景到一套可扩展的协调决策引擎

场景一（会议协调）和场景二（时间申请仲裁）表面问题不同，但拆开后是同一套骨架在跑：分级规则 → 自动化匹配/排序 → 有效性/优先级判定 → 明确的升级边界。这个发现是体系化的起点——**不为每个新流程重做一套系统，而是把这套骨架抽成一个可复用的引擎，新流程只需接入自己的规则，其余复用**。

### 六层引擎架构

```
① 资源/日程读取层     读取参与方日程、占用状态、历史数据 — 全体系共用
② 候选生成层           候选时段 / 候选排序方案 — 每个模块专属
③ 规则评分与判定层     角色权重、优先级评分、有效性阈值 — 每个模块专属，规则表可配置
④ 沟通编排层           分级触达 + 打扰成本感知，跨模块合并提醒 — 全体系共用
⑤ 升级仲裁层           统一收件箱，汇总所有模块"需要人判断"的事项 — 体系枢纽
⑥ 看板渲染层           读同一份状态数据，套用统一视觉模板渲染 — 全体系共用
```

新增一个工作流（例如"会议室资源分配"“临时任务插单排序”），只需要新写②候选怎么生成、③规则表长什么样，其余四层直接复用。⑤升级仲裁层是整个体系的枢纽——它定义了一套所有模块必须遵守的"契约"，新模块只要按这个契约抛出请求，就能自动接入。

### 统一升级请求格式（Escalation Item）

这是让体系"接得起来"的关键约定，任何模块的升级请求都必须符合同一结构：

| 字段 | 说明 |
|---|---|
| `id` | 唯一编号 |
| `source_module` | 来源模块（会议协调 / 时间申请仲裁 / 未来新模块） |
| `urgency` | 紧急度：高 / 中 / 低 |
| `summary` | 一句话摘要，人一眼能看懂在问什么 |
| `requested_decision` | 需要人做的具体决策类型 |
| `candidates` | 可选方案列表，每个方案带优劣说明，供人直接挑选 |
| `deadline` | 决策截止时间 |
| `status` | 待处理 / 已处理 / 已过期 |
| `resolution` | 处理结果、处理人、处理时间——处理后回填，用于沉淀学习 |

`resolution` 字段的沉淀尤其关键：每一次人工决策的结果都被记录下来，反过来修正③规则评分层——比如系统发现"这类冲突协调者总是选择合并沟通"，下次可以直接把这个倾向作为候选方案的默认排序，而不是每次从零升级。这是让体系"越用越聪明"的机制。

### 支持材料

- **引擎架构图**：六层结构可视化，标注哪些层通用、哪些层需按模块新写；
- **统一升级仲裁收件箱原型**：汇总各模块的升级事项，按统一卡片格式展示，支持按来源模块/紧急度筛选，点击查看候选方案与决策依据，已处理事项展示其沉淀下来的规则；
- **控制台**：所有模块的统一入口，支持拖拽调整模块展示顺序。

## 九、体系验证：五个新模块的接入

场景一、二验证了引擎本身能跑通，但只用两个高度相似的"资源仲裁"场景来验证一套体系，说服力有限——如果新场景恰好都长得像"抢时间"，规则复用听起来更像巧合而非设计。因此接下来按同一套引擎，接入了五个来源、结构都不相同的真实工作场景，用来检验这套架构是否真的具备可扩展性，也检验它是否会被过度套用。

### 场景三：经费管理台账

最初按"稀缺资源仲裁"的思路去设计，但深入了解实际情况后发现**判断错了问题性质**——这里的预算总额固定、类别为人为划分的统计口径，不存在"多方竞争同一份资源"的问题,核心诉求其实是记录与预警，而非排序仲裁。据此重新设计：②候选生成层、③规则排序层基本不需要，只保留一个"总预算即将用尽"的预警判断和"审批驳回留痕"，其余是台账记录与看板展示。这个调整过程本身也说明了一件事：**接入新模块前，先判断它是否真的属于"仲裁类"问题,比套用现成框架更重要**。

### 场景四：新员工与导师管理

同样出现了一次设计假设被推翻：最初以为"导师分配"是一个可以用②候选生成 + ③规则匹配来做的问题，但实际了解到导师人选由业务组 PL/LM 前置决议，不需要、也不适合由 AI 介入判断。因此这个模块真正落在引擎里的判断点,只有转正评级这一处——按"试用期连续两次 EXCELLENT + 答辩得分"的规则自动判定良好/优秀，边界情况（分数达标但未满足门槛）标记待人工复核，不自动下结论。

### 场景五：信息收集与反馈看板

这是五个新模块里跟场景一结构最接近的一个——多任务并行、催办轮次、临近截止未达标自动升级,几乎是会议协调模块的\"多任务版\"。同时验证了另一件事：最初描述里\"培训活动的课程反馈收集\"看起来是培训知识管理的一部分，但拆开来看，它和这个模块是同一种模式（多人、并行、催收、升级），因此直接并入本模块，而不是重复做一遍。

### 场景六：经理日程管理

核心判断点是"这个会议是否仅为抄送，可以自动隐藏"——按"经理为抄送非主送"或"经理已点击谢绝"两条规则自动判定，处理方式是隐藏但保留记录，而非删除。同时加入了一个规则之外的安全边界：若某类会议历史上曾被临时要求发言,即便本次表面符合过滤条件，也标记待确认而非直接隐藏，避免规则判断过于机械。

### 场景七：员工生日关怀

五个模块里唯一一个**几乎不需要仲裁层**的模块——本质是定时触发的自动化任务，只复用了④沟通编排层和⑥看板渲染层，②③⑤三层基本用不上。唯一的例外场景是员工请假或已离职时，群祝福是否合适需要人工确认。这个模块刻意做得很轻，是为了如实呈现它在体系里的定位，而不是为了体系统一强行给它加不必要的判断逻辑。

### 五个模块合起来验证了什么

- 引擎的六层结构确实可复用，但**不是每个模块都要用满六层**——场景三、四、七都只用到了其中两三层，这恰恰说明这套架构设计得足够灵活，能按问题的真实复杂度做裁剪，而不是把简单问题也套上一整套仲裁流程；
- 真正决定一个新场景该怎么设计的，是先判断"这里有没有资源竞争/边界模糊的判断"，而不是先假设它一定适合套用现成模式——场景三、四都经历过一次设计假设被推翻再重新定位的过程,这个纠偏能力本身也是这套方法论的一部分；
- 统一升级仲裁收件箱在接入全部七个模块后，无需修改自身结构，只是新增了对应的来源标签和筛选项——验证了当初"⑤层是体系唯一契约点"的设计判断。

---

## 十、这个案例想说明什么

这套方案没有依赖任何前沿技术，核心价值在于**把一线真实业务中的隐性规则（角色权重、有效性判定、升级边界）显性化、结构化，转化为可配置的系统逻辑**。这正是大量"通用型 AI Agent 产品"目前缺失、也最难从纯技术背景补齐的部分——它需要真正在一线处理过这类协调工作的人来设计。

从两个场景扩展到七个模块的过程中，也验证了一件比"引擎能跑"更重要的事：**判断一个新场景是否需要仲裁、需要多少层判断逻辑，比机械套用现成框架更关键**。经费管理、导师分配这两次"推翻重来"的设计调整，恰恰是这套方法论里最真实的部分——它不是一套只会往一个模子里套的模板,而是一套先诊断问题性质、再决定该用几层的判断框架。

这也是我理解的、自己在 AI 领域可以提供的价值：不是重新造一个执行工具，而是设计"AI 该在哪里自主、该在哪里交还给人"的规则本身。
